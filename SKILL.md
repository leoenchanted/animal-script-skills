# 动物科普短视频脚本 Skill

## 触发与输出

当用户输入包含“想了解”“介绍一下”“写一篇关于”“科普文案”“视频脚本”“分镜”“生图提示词”等表达，且主题涉及动物或生物时，使用本 skill。

默认输出必须包含两种形式：

1. Markdown 视频脚本：视频设定、分镜表、中英文 AI 生图提示词、完整口播稿、剪辑建议、事实参考来源。
2. 可编辑 HTML 分镜工作台：保存为本地 `.html` 文件，供用户逐镜头编辑和复制提示词。

不要等用户主动要求 HTML；只要用户请求生成动物脚本、分镜或生图提示词，就同时交付 Markdown 和 HTML 工作台。

## 核心目标

生成适合短视频制作的动物科普脚本，而不是普通文章。输出必须帮助用户直接进入视频制作流程：

1. 先搜索权威资料，保证动物事实准确。
2. 再生成 60-90 秒短视频解说脚本。
3. 将脚本拆成 10-15 个镜头。
4. 每个镜头提供对应画面描述、中文 AI 生图提示词、英文 AI 生图提示词和动效建议。
5. 默认视觉方向为卡通科普风，不直接提供网络图片链接，除非用户明确要求真实图片素材。
6. 同步生成一个可编辑 HTML 分镜工作台，并完成静态验证。

## 执行流程

### 第一步：确认输入

从用户请求中提取：

- 动物名称
- 视频平台或用途（如抖音、视频号、B站、小红书）
- 期望时长（未指定时默认 60-90 秒）
- 视觉风格（未指定时默认卡通科普风）
- 目标受众（未指定时默认大众科普）

如果用户只给出动物名称，也直接执行，不要反复追问。

### 第二步：检查模板与叙事风格样本

优先级：

1. 如果项目里存在 `.claude/templates/animal-video-template.md` 且有内容，优先按用户项目模板的输出结构生成。
2. 读取本 skill 的 `template-default.md`，作为默认输出结构参考。
3. 查看 `examples/style-templates/` 下的 `.txt` 文件，作为叙事风格样本库。

叙事风格样本只用于学习：

- 开头设问方式
- 语气语调
- 短句节奏
- 悬念推进
- 情绪转折
- 故事结构
- 结尾余味

不要把样本中的动物、数值、生态关系、年代、捕猎细节当作事实来源。生成新脚本时，必须用第三步搜索到的权威事实重新构建内容。

如果样本库里有多个 `.txt` 文件，先快速判断每个样本的主题、叙事类型和语气，再选择 1-3 个最适合当前动物与用户需求的样本融合风格。不要依赖文件名判断风格，要读内容开头和关键段落。

读取样本后，先在内部抽取一张“风格卡片”：

- 开头钩子：疑问、反差、危机、画面降临等
- 叙事视角：旁白、跟随主角、纪录片式观察等
- 语气质感：冷峻、紧张、悲壮、轻快、温柔等
- 节奏方式：短句密度、悬念间隔、转折位置
- 冲突类型：环境压力、捕食压力、成长压力、同类竞争、人类影响等
- 结尾方式：反转、余味、科普总结、情绪落点

最终脚本只能继承这张风格卡片，不要复制样本文字、桥段或具体设定。不要机械套用单一模板；要根据真实动物事实决定故事节奏。如果动物事实不支持强戏剧冲突，就降低戏剧强度，改用更适合的科普叙事。

当用户请求类似“如果把某种动物放到某个现代环境”“史前动物复活”“能不能存活”“生态位巅峰”“穿越到现代”等设定时，优先参考生存推演类样本的开头、节奏和戏剧推进方式，同时明确区分“已证实事实”和“基于事实的合理推演”。

### 第三步：权威资料搜索

围绕动物主题进行多维度搜索。事实必须来自可核验来源，不要编造数据。

优先来源：

1. IUCN Red List
2. WWF
3. National Geographic
4. Britannica
5. Smithsonian / San Diego Zoo / AZA 等动物机构
6. 学术机构、博物馆、政府保护组织

搜索维度：

| 维度 | 目标信息 |
|:---|:---|
| 基本档案 | 学名、分类、体型、体重、寿命 |
| 外形特征 | 毛色、纹路、身体构造、辨识点 |
| 栖息地 | 地理分布、典型环境、气候条件 |
| 生活习性 | 活动时间、社交方式、领地行为 |
| 饮食策略 | 食物类型、捕食方式、取食技巧 |
| 繁殖育幼 | 求偶、产仔/产卵、育幼方式 |
| 生存技能 | 特殊能力、防御方式、适应机制 |
| 保护现状 | IUCN 等级、种群趋势、主要威胁 |
| 反差亮点 | 适合短视频开场的惊讶点 |
| 冷知识 | 适合中段节奏变化的趣味事实 |

### 第四步：生成视频策划

输出脚本前先给出视频基础设定：

- 视频标题
- 一句话爆点
- 视频时长
- 目标平台
- 叙事节奏
- 统一视觉风格
- 主角角色设定
- 参考的叙事风格样本（如有）

统一角色设定必须让每张 AI 图保持一致，包括：

- 动物外形关键词
- 卡通化程度
- 色彩氛围
- 光影质感
- 画幅比例
- 负面限制（无文字、无水印、不要真实照片风等）

### 第五步：生成分镜表

默认输出 10-15 个镜头。每个镜头必须包含：

| 字段 | 要求 |
|:---|:---|
| 镜头 | 两位数编号，如 01 |
| 时间 | 例如 0-5s |
| 解说词 | 可直接配音的中文口播 |
| 画面内容 | 视频中应出现的画面 |
| 中文生图提示词 | 可复制到国产 AI 生图工具 |
| English Prompt | 可复制到 Midjourney / DALL-E / SD |
| 动效建议 | 推拉摇移、字幕节奏、转场、局部动画 |
| 事实依据 | 简短标注来自哪个来源或哪类资料 |

提示词要求：

- 使用卡通科普、动画短片、儿童科普绘本、3D 卡通或干净插画等风格词。
- 每条提示词都要包含主体、场景、动作、情绪、构图、光影、比例。
- 默认比例为 16:9。
- 默认加入限制：无文字、无水印、无 logo、不要真实照片风。
- 不要要求生成真实摄影作品，除非用户明确指定。
- 不要让提示词包含虚假的文字标签或难以稳定生成的小字。

如果采用叙事风格样本，先写出一版完整故事口播稿，再拆成分镜表。分镜表要服务于故事推进，而不是把资料条目平均铺开。开头应有强设问或强画面，中段用事实制造压力和转折，结尾要有情绪收束或认知反转。

### 第六步：输出完整口播稿

在分镜表后输出一版连续口播稿，方便用户直接录音。

要求：

- 语言自然，有短视频节奏。
- 数据表达清楚但不过度堆砌。
- 重要事实与分镜表一致。
- 结尾有评论互动或情绪收束。

### 第七步：输出制作清单

最后输出：

- AI 生图清单
- 动效剪辑建议
- 配音语气建议
- 背景音乐建议
- 事实参考来源

事实来源和 AI 画面提示词必须分开，不要把 AI 图当作真实图片来源。

### 第八步：生成 HTML 分镜工作台

默认创建一个独立 HTML 文件，文件名使用英文短横线，例如：

`snow-leopard-storyboard.html`

HTML 必须是可直接打开的自包含页面，除非项目已有明确前端栈，否则不要依赖构建工具。页面必须包含：

- 优先以本 skill 的 `template-storyboard.html` 为基础，替换其中 `project` 数据对象和必要主题变量。
- 顶部展示标题、动物、时长、镜头数、视觉风格。
- 主区域展示镜头卡片，而不是普通文章或纯表格。
- 每张卡片展示镜头编号、时间、解说词、画面内容、中文提示词、英文 Prompt、动效建议、事实依据。
- 解说词、画面内容、提示词、口播稿使用可编辑 textarea。
- 提供复制单条中文提示词、复制单条英文 Prompt、复制全部提示词、复制完整口播稿的按钮。
- 提供统一角色设定、制作建议、事实来源链接。
- 所有 AI 生图提示词和事实来源分开展示。

HTML 设计必须内置完成，不依赖其他 skill：

- 做成视频制作工作台，而不是普通博客文章。
- 使用清晰的信息密度、稳定网格、响应式布局，桌面和移动端都不能重叠或溢出。
- 卡片圆角不超过 8px；不要把卡片套在卡片里。
- 按钮、复制控件和编辑区要有明确 hover/focus 状态。
- 不要用纯装饰渐变球、空洞大 hero、营销落地页式布局。
- 颜色、字体、空间和动效要贴合动物主题，但不能牺牲可读性。
- 页面中不要出现介绍“如何使用本页面”的大段说明文字；功能通过控件和布局自己表达。

### 第九步：静态验证 HTML

生成 HTML 后做轻量静态验证，避免引入额外依赖或网络安装。验证至少覆盖：

- HTML 文件已创建，文件名清晰，后缀为 `.html`。
- 如使用 `template-storyboard.html`，确认已经替换占位数据，不要把 `【动物名称】` 这类占位符留在最终文件里。
- 页面包含标题、镜头卡片容器、统一角色设定、完整口播稿、制作建议和事实来源区。
- HTML 内脚本语法可解析，例如提取 `<script>` 内容后用 Node `new Function(script)` 检查。
- 镜头数量与 Markdown 分镜一致，编号从 `01` 连续到最后一镜。
- 每个镜头都包含解说词、画面内容、中文提示词、英文 Prompt、动效建议和事实依据。
- 复制单条提示词、复制全部提示词、复制口播稿等关键按钮或 `data-*` 标记存在。
- Markdown 脚本与 HTML 工作台内容一致，没有漏镜头、漏提示词或漏来源。

如发现问题，先修复 HTML，再重新做静态验证。最终回复用户时说明 HTML 文件路径和静态验证结果。

## 质量标准

- 所有动物事实必须可追溯到来源。
- 叙事风格样本不是事实来源，只能借鉴语气、节奏、结构和开头感觉。
- 使用推演型写法时，必须把推演建立在已查证事实上，不要把想象当成确定结论。
- 不要为了戏剧效果编造体型、速度、寿命、分布或保护等级。
- 不要把多个相近物种的数据混在一起，除非明确说明是在讲整个类群。
- 不要输出真实图片 URL 作为主要素材；本 skill 的主要交付物是 AI 生图提示词。
- 画面提示词要服务于视频节奏，每段解说都应有对应画面。
- HTML 工作台必须和 Markdown 脚本内容一致，不能漏镜头、漏提示词或漏来源。
